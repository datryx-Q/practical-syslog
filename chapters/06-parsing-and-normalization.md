# 06 – Parsing and Normalization

Logs are only useful if they can be understood.

Parsing and normalization attempt to turn raw log messages into structured data that can be searched, correlated, and analyzed.

This sounds straightforward.

It is not.

---

## The Core Problem

Logs are generated by:
- Different applications
- Different operating systems
- Different vendors
- Different teams

Each produces data differently.

Example:

```plaintext
Jan 12 10:00:00 web01 sshd[1234]: Failed password for root
2026-01-12T10:00:00Z firewall01 action=deny src=10.0.0.1
{"timestamp":"2026-01-12T10:00:00Z","event":"login"}
```
All three represent events.

None share the same structure.

## What Parsing Does

Parsing extracts structure from raw text.

Example:

```plaintext
Jan 12 10:00:00 web01 sshd[1234]: Failed password for root
```

Becomes:

```
| Field     | Value                    |
|-----------|--------------------------|
| timestamp | Jan 12 10:00:00         |
| hostname  | web01                   |
| program   | sshd                    |
| pid       | 1234                    |
| message   | Failed password for root |
```

This enables:

- Searching
- Filtering
- Correlation
- Alerting

## What Normalization Does

Normalization standardizes fields across different log formats.

Example:

Vendor A |	Vendor B | Normalized
src_ip | source | source_ip
user | username | user
sev | priority | severity

Without normalization:

- Queries become inconsistent
- Correlation becomes difficult
- Detection rules become fragile

## The Hidden Assumption

Most parsing systems assume: Logs are predictable

Reality:

- Formats change
- Ven
- dors violate standards
- Applications emit inconsistent data
- Developers modify log structures without warning

Parsing systems fail constantly.

## Structured vs Unstructured Logs
### Unstructured Logs

Traditional syslog:

```plaintext
Jan 12 10:00:00 app01 app[5000]: user login failed
```

Human-readable, but difficult to parse reliably.

## Structured Logs

JSON example:

```json
{
  "timestamp": "2026-01-12T10:00:00Z",
  "event": "login_failed",
  "user": "admin"
}
```
Machine-friendly and easier to normalize.

## Structured Logging Is Not a Complete Solution

Even structured logs have problems:

- Inconsistent field names
- Nested formats
- Invalid JSON
- Partial serialization

Example:

```
{"event":"login","user":"admin"
```

Missing one character can break an entire parser.

## Parsing Failure Modes
### 1. Complete Failure

The parser rejects the message entirely.

Result:

- Message dropped
- Message rerouted
- Processing stops

### 2. Partial Failure

Some fields parse correctly, others do not.

Result:

- Incomplete data
- Incorrect searches
- Broken dashboards

### 3. Silent Misparse (Worst Case)

The parser succeeds—but incorrectly.

Example:

- Wrong timestamp extracted
- Incorrect hostname
- Truncated message

This is dangerous because:

The system appears healthy while data quality degrades

Why `malformed.log` Matters

This dataset intentionally contains:

- Missing fields
- Broken timestamps
- Corrupted payloads
- Mixed formats

Its purpose is not to break the parser.

Its purpose is to reveal:

How the parser behaves when reality does not match assumptions

## Regex Is Fragile

Many pipelines rely heavily on regex parsing.

Example:

```
^(\w+\s+\d+\s+\d+:\d+:\d+)\s+(\S+)\s+(\S+):
```

This works until:

- Extra spaces appear
- Formats change
- Fields disappear

Regex-based parsing becomes difficult to maintain at scale.

## Normalization Tradeoffs

Normalization improves consistency, but introduces:

Additional processing overhead
Increased complexity
Mapping maintenance

Over-normalization can:

- Hide useful vendor-specific details
- Introduce incorrect assumptions

## Parsing at Scale

Under high throughput:

- Parsing becomes CPU-intensive
- Latency increases
- Queues grow

In many pipelines:

Parsing becomes the bottleneck
